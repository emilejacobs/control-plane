# Glossary

Terms used throughout the design documents.

| Term | Meaning |
|---|---|
| **CP** | Control Plane. This project — the AWS-hosted system. |
| **Edge device** | Any uKnomi-managed hardware at a client site (Mac Mini, Raspberry Pi, Radxa Rock). |
| **Edge UI** | The on-device companion to CP. Today: a Flask app at `mac-mini-rollout/webui/`. After the [ADR-029](./adr/0029-edge-ui-rework-scope.md) / [ADR-030](./adr/0030-edge-ui-per-feature-surface.md) rework: a Next.js app with exactly two pages — **Camera live preview** and **Audio test** — running on the tailnet. No auth, no CP credentials, no persistent state. Hardware-bound only. Formerly named "Talon". |
| **Camera live preview** | The MJPEG stream from an on-device ffmpeg pipe over a camera's RTSP URL, served by Edge UI at `http://<host>:5051/preview/<camera_id>` (v1 per [ADR-032](./adr/0032-edge-ui-v1-stack-port-tls.md); `https://<host>.<tailnet>/...` remains the long-term target in [ADR-030](./adr/0030-edge-ui-per-feature-surface.md) § 1). The dashboard's `edgePreviewURL` prefers `device.tailscale_name` over `device.hostname` for `<host>` (issue #14) — `os.Hostname()` and the Tailscale-resolvable name can diverge, so the heartbeat-published name is authoritative for MagicDNS resolution. When `device.lan_ip` is also known, the panel exposes a secondary "Copy LAN URL" affordance — the LAN-IP fallback hint deferred from issue #4 per ADR-032. Used by an operator clicking "Verify angle" in CP. No auth (tailnet-as-perimeter). |
| **Network scan** | An on-demand LAN scan run by the agent in response to a `network.scan` cmd from CP. Output is a list of candidate IPs with MAC, vendor (OUI lookup), and open camera-relevant ports. Used during install to find cameras. |
| **Capture** | A device-produced binary artifact stored in S3 and indexed in CP. Three kinds today: **Snapshot**, **Audio recording**, **Transcript**. Upload via CP-vended pre-signed PUT URL; agent is the upload gateway for all three kinds. |
| **Snapshot** | A capture of a camera's RTSP frame, JPEG. Produced by the agent — scheduled weekly per camera plus on-demand via `camera.snapshot` cmd. 90-day S3 retention. |
| **Audio recording** | A capture of microphone audio, WAV. Produced by Edge UI's audio test page; agent picks it up via `fsnotify` and uploads. On-demand only. No retention policy (kept forever). |
| **Transcript** | The text output of running Whisper against an Audio recording, JSON+TXT. Produced by Edge UI's audio test page during install for audio quality QA. No retention policy. |
| **Webhook endpoint registry** | The CP-wide table of named webhook endpoints (`transcriber-prod`, `transcriber-dev`, etc.). Per-device PR config references entries by name; CP resolves names → URLs at `pr.config.update` push time. First instance of the [ADR-031](./adr/0031-webhook-endpoint-registry.md) fleet-wide-config pattern. |
| **Camera (record)** | A per-device camera-inventory row stored in `device_cameras`. Server-assigned `camera_id` of the form `cam1`, `cam2`, … per device; fields are `label`, `rtsp_url`, `is_lpr`. CP owns the table; agent's local `cameras.json` is a downstream copy synced via the `cameras.update` cmd. At most one `is_lpr=true` per device (DB partial unique index). |
| **`cameras.update`** | Downward CP→agent cmd (issue #2) carrying the full cameras list for a device. Wire shape `{cameras: [Camera...]}`; field-whitelisted per the ADR-028 protective stance. Agent applies via atomic write to the local cameras file; cp-ingest stamps `devices.cameras_last_applied_*` on the ACK. |
| **`network.scan`** | Downward CP→agent cmd (issue #3) triggering a LAN scan on the device. Optional `cidr` field overrides the auto-detected primary IPv4 /24. Agent shells out to `nmap`, enriches results with OUI vendor lookup + camera-port filter, and ACKs with `{hosts: [{ip, mac, vendor, open_ports}]}` on `cmd-result`. cp-ingest writes the result into `device_network_scans` keyed by `correlation_id`. Unsigned per ADR-028 (extended); the sixth such handler. |
| **`device_network_scans`** | The per-request row store for network-scan flows. PK `correlation_id`; FK to `devices` with `ON DELETE CASCADE`; status walks `pending → done | error`; `result` JSONB stays NULL until the agent's ACK lands. Same lifecycle shape as `device_log_tails`; the cp-ingest sweeper reaps rows at 24h. |
| **Agent** | `uknomi-agent`, the Go binary on every device that talks to the CP via MQTT. |
| **Tailnet** | The uKnomi Tailscale network. All edge devices and the CP's Tailscale subnet router are members. |
| **Subnet router** | A Tailscale node that advertises a route, allowing other tailnet members to reach hosts behind it. The CP runs one as a Fargate task to give the API service tailnet access without enrolling clients. |
| **Device shadow** | An IoT Core feature representing desired and reported state of a thing as JSON documents. CP uses it for service-state tracking. |
| **Bootstrap key** | A static shared secret bundled into the device install package; the install script presents it to `POST /enrollments`. Static-key approach per ADR-017, which superseded the short-lived per-device S3 token of ADR-014. |
| **Enrollment** | The one-time process of registering a new device with the CP and issuing its mTLS cert. Happens *inside* **Provision** — the install binary's `POST /enrollments` call. |
| **Provision** | The one-shot, device-side install step: host configuration + the uniform software set (Homebrew formulae, Colima, ffmpeg, whisper model, the ALPR image) + laying down the agent/supervisor + **Enrollment**. Identical on every Mac — carries no site identity. Run by `uknomi-agent install`, triggered by **Mosyle**. Retires the former `mac-mini-rollout` "Phase 1 / HQ staging" step. *Not* to be confused with the CP project's rollout **Phases 1–3** — a different axis. |
| **Assign** | The operator action of binding an enrolled-but-unassigned device to a **Site** in the CP dashboard (sets `devices.site_id`; the #64 rollout site-picker surface). A device is **unassigned** between Provision and Assign — online and idle with a NULL `site_id`. |
| **Commission** | The CP-driven step that brings an **Assign**ed device into service by pushing everything site-specific to it: `cameras.update`, the ALPR `config.ini` + per-device license (which starts the container and is the point the license is consumed), the Tailscale auth key (to join the tailnet), the service allow-list, and snapshot cadence. Replaces the former on-site "Phase 2 activation" — _avoid_ "activate" for this, which the retired two-phase model used for the on-site tech turning a device on for its store. |
| **Signed command** | A command payload signed with the CP's Ed25519 key (held in KMS) before being sent to a device. The agent verifies the signature before executing. |
| **Client** | A uKnomi customer organization (e.g. "Rao"). May operate one or more brands (e.g. Burger King and Dunkin Donuts franchises) and owns one or more sites. CP's `clients` table; canonical source is the upstream HTTP API at `api.uknomi.com` (mirrored daily per [ADR-033](./adr/0033-clients-sites-taxonomy-sync.md)). |
| **Site** | A physical client location, operated under one Brand (e.g. Rao's Burger King store at Mesa). CP's `sites` table; `client_id` FKs to the Client; `brand_name` + `brand_external_id` capture Brand as metadata. Mirrored from the upstream API per [ADR-033](./adr/0033-clients-sites-taxonomy-sync.md). One Client may have multiple sites; sites are unique per Brand (one physical location = one Site per Brand). |
| **Brand** | The franchise system under which a Site operates (e.g. "Burger King", "Dunkin Donuts"). CP does **not** model Brand as a hierarchy level — Brand is captured per Site as metadata (`sites.brand_name` + `sites.brand_external_id`) for display and traceability. The upstream API at `api.uknomi.com` exposes Brand as its top-level navigation key (`GET /brand` then `GET /brand/{id}/store`); the [taxonomy-sync](#) job iterates active brands to enumerate sites but doesn't materialize a `brands` table in CP. A Client can operate multiple Brands; a Brand spans multiple Clients (different franchisees of the same chain). [ADR-033](./adr/0033-clients-sites-taxonomy-sync.md). |
| **`taxonomy-sync`** | The Fargate scheduled task (`cmd/taxonomy-sync`) that mirrors clients + sites from the upstream `api.uknomi.com` HTTP API into CP's local `clients` + `sites` tables. Runs daily at 00:05 UTC via EventBridge; can also be triggered ad-hoc by a staff operator via the Settings page "Force sync now" button (cp-api `POST /taxonomy/sync` → `ecs:RunTask` on the same task def). Postgres advisory lock prevents concurrent runs. Soft-deletes via dual signal (API `active=false` OR absent-from-sync). [ADR-033](./adr/0033-clients-sites-taxonomy-sync.md). |
| **`external_id`** | The upstream API's primary key for a Client or Site, stored on the local mirror row (`clients.external_id` / `sites.external_id`, both `UNIQUE`). The upsert key for [taxonomy-sync](#); also the identifier operators use when referring to specific sites in conversation. CP's local UUID `id` stays as the FK referent for `devices.site_id` and `operator_sites.site_id`. [ADR-033](./adr/0033-clients-sites-taxonomy-sync.md). |
| **`active` (clients/sites)** | The local soft-delete flag set by [taxonomy-sync](#). Dual signal: the upstream API returned the record with `active=false` OR the record was absent from this sync's responses (its brand went inactive, or it was hard-removed upstream). Pickers filter to `active=true`; display surfaces render inactive entities with an "Inactive" badge so operators see staleness rather than silent disappearance. Never hard-deleted — preserves FK integrity for `devices.site_id` + `operator_sites.site_id` and supports free reactivation. [ADR-033](./adr/0033-clients-sites-taxonomy-sync.md). |
| **`last_synced_at` (clients/sites)** | The timestamp of the most recent taxonomy-sync run that observed this record. Used by the post-sync sweep (`UPDATE … SET active=false WHERE last_synced_at < $sync_start_time`) to flip absent-from-current-sync rows to inactive. Sync-start time (not sync-end) is the threshold, so rows added mid-run survive the sweep and get observed next run. [ADR-033](./adr/0033-clients-sites-taxonomy-sync.md). |
| **Asset number** | A fleet-tracking identifier assigned to each device at install time. Stored on `devices.asset_number` (migration 014); NULL until install-module 11 ships it. *Not* the leading numeric prefix on hostnames like `07-eegees-mesa-macmini` — that's something else. Rendered on the per-device Deployment card; "Unassigned" when NULL. |
| **Operator (staff)** | A uKnomi internal user with full fleet access. Authenticates with local credentials — password + TOTP (ADR-010, which dropped Entra ID); carries the `'*'` site allowlist via the `is_staff` flag. |
| **Operator (local)** | A future field-operator user with a local CP account, scoped to specific sites. |
| **Site allowlist** | The set of sites an operator may see. Staff hold the full fleet implicitly via `is_staff`; a non-staff operator's allowlist is the explicit `operator_sites` grants. Enforced on every device read through the `authz` module's `ScopedDeviceQuery` helper. |
| **Operator management** | The staff-only `/operators` surface (#16) to list / add / edit / deactivate CP operators without DB access. The `operators` Go package (distinct from `authn`, which owns the auth path) backs `GET/POST/PUT /operators` + deactivate/reactivate, all behind auth + TOTP + the staff gate and audited. |
| **Temp password** | A system-generated, CSPRNG one-time password issued when a staff admin **creates** an operator or **resets** its password (no SMTP — relayed out-of-band). Returned in the API response exactly once; persisted only as an Argon2id hash, never logged. Arms *must-change-password*. |
| **Must-change-password** | The `operators.must_change_password` flag set by create / password-reset. While armed, login signals it (alongside `requires_totp_enrollment`) and the `RequirePasswordChanged` gate 403s every protected route — including TOTP enrollment — except the constrained `POST /auth/password` set-new-password path. Onboarding order: set password → enroll TOTP → access. Cleared once the operator sets a new password. |
| **Deactivated operator** | An operator soft-deleted via `operators.deactivated_at` (#16). Cannot authenticate (login returns invalid-credentials, state not revealed), but the row and its audit-log references are preserved; reactivation clears the flag. Chosen over hard delete to keep the audit trail intact. |
| **Mosyle** | Apple MDM provider used for Mac auto-enrollment. Not in the CP runtime path; only triggers the install script. |
| **Heartbeat** | A small message the agent publishes to `devices/{id}/telemetry` every 30s. The ingest worker uses it to update the device's `last_seen` in Postgres. Issue #14 added three optional network fields — `lan_ip`, `tailscale_ip`, `tailscale_name` — published per-tick when their detectors return non-empty and persisted to the matching `devices.*` columns via conditional UPDATE (last-wins on present fields; absent fields preserve the stored value). |
| **`lan_ip`** | The device's primary RFC1918 IPv4 address (e.g. `192.168.54.215`). Detected by the agent's `PrimaryRFC1918Addr` per-heartbeat (RFC1918 filter shared with `subnetCandidate`; 100.64.0.0/10 CGNAT excluded so a tailnet address never leaks here). Persisted on `devices.lan_ip` (migration 018; nullable). Used by the dashboard's "Copy LAN URL" affordance — when set, the CamerasPanel renders a secondary deep-link `http://<lan_ip>:5051/preview/<cam_id>` alongside Verify angle. The LAN-IP fallback hint from [ADR-030](./adr/0030-edge-ui-per-feature-surface.md) § 1, deferred per [ADR-032](./adr/0032-edge-ui-v1-stack-port-tls.md) until telemetry shipped it (issue #14). |
| **`tailscale_ip`** | The device's CGNAT (100.64.0.0/10) IPv4 address. Detected by the agent's `PrimaryTailscaleAddr` per-heartbeat (utun\* on macOS / `tailscale0` on Linux + CGNAT range filter; same heuristic as Edge UI's `DetectListenAddrs`). Persisted on `devices.tailscale_ip` (migration 018; nullable — empty for non-tailnet devices and pre-rollout agents). |
| **`tailscale_name`** | The device's Tailscale MagicDNS name (e.g. `07-eegees-store54-macmini.tailnet.ts.net`). Resolved by `ResolveTailscaleName` shelling out to `tailscale status --json` and reading `Self.DNSName` with the trailing dot stripped; a missing binary or logged-out tailscaled degrades silently to NULL so the heartbeat never fails on dev/non-tailnet devices. Persisted on `devices.tailscale_name` (migration 018; nullable). The dashboard's `edgePreviewURL` prefers this over `device.hostname` to handle the bench-Mac drift case (`os.Hostname()` ≠ MagicDNS view of the node). |
| **`boot_time`** | The device's system boot instant, read once at agent start via `kern.boottime` (`unix.SysctlTimeval`, no subprocess) and reported on every heartbeat (RFC3339). The reboot-vs-blip discriminator (#157, PRD `.scratch/offline-reason-tracking`): cp-ingest compares it to the device's stored `devices.last_boot_time` — a **change** means the device rebooted; **unchanged** across an offline→online window means a network/MQTT blip. macOS-only; omitted on other platforms (the fleet is Macs). Persisted on `devices.last_boot_time` (migration 032; nullable until a rolled agent reports it). |
| **`last_shutdown_cause`** | The macOS *previous shutdown cause* that rides with `boot_time` (#157): an integer code parsed from `log show … "Previous shutdown cause"` once at agent start, mapped to a label by the agent's `bootinfo` reader — `5` → clean restart, `3` → forced/hung, `0` → power loss, negative codes → thermal/watchdog/power/panic (best-effort, refined against real logs), anything unmapped passed through verbatim as `unknown (code N)`. Persisted on `devices.last_shutdown_cause` (+ raw `last_shutdown_cause_code`, migration 032). Answers *why* a reboot happened. |
| **`device_reboots`** | Per-device reboot history (migration 032): one row per distinct `boot_time` observed (device_id, boot_time, shutdown_cause, code, detected_at), keyed unique on `(device_id, boot_time)` so a replayed heartbeat is a no-op. Written by `Registry.RecordBootInfo` on each boot_time change (including first contact, which is recorded but not alerted as a reboot). Drives the device-page reboot list (#159 — surfaced via `Registry.ListReboots` + `recent_reboots` on `GET /devices/{id}`, rendered by the device page's *Restart History* card) and the offline-reason label on recovery (#158, which reads rows inside the offline window). |
| **Offline debounce** | A grace window before a `device_offline` alert fires: the notification reconciler holds back a newly-offline signal until the device has been continuously offline for `notifications.offline_grace_seconds` (default **180s**, tunable in Settings → Notifications, `0` disables). A sub-grace network/MQTT blip recovers before the window and never opens an OFFLINE + recovered pair — killing the Teams-channel flap noise — while a sustained outage (and reboots, which take minutes) still alerts, ~grace later. Offline-kind only; `service_stopped`/`probe_red` fire immediately. Offline-duration comes from `UnhealthySignal.Since` (= `devices.presence_changed_at`). Orthogonal to and complements *Offline reason* (debounce suppresses noise; reason explains the survivors). |
| **Offline reason** | Why a fleet device went offline: a **reboot** (with its `last_shutdown_cause`) when `boot_time` changed across the offline→online window, or a **network/MQTT blip** when it didn't (#157 pipeline). The device page surfaces last boot + last shutdown cause + a Restart History list (#159). Inferred entirely from `boot_time` deltas, so it's robust even when CP misses the exact disconnect event (e.g. during a cp-ingest deploy). Distinct from AWS IoT `disconnectReason` (Tier 0, deferred). On recovery, `Registry.OfflineReason(deviceID, since, until)` derives it from `device_reboots` over the offline window `[opened_at, resolved_at]` and the notification reconciler tags the resolved `AlertEvent.Reason`; the email + Teams recovery line renders it (#158). A device that never reported boot info (pre-#157 agent) yields no reason — rendered as nothing, never a false "network blip". |
| **`host_net_pressure`** | A health probe ([ADR-040](./adr/0040-cp-scored-metric-health-probes.md)) reporting host TCP/ephemeral-port pressure — distinct local ephemeral ports in use as a % of the pool, plus the `CLOSE_WAIT` count (a socket-leak fingerprint). Born from the 2026-06-26 store13/mesa ephemeral-port-exhaustion incident, where a wedged kernel TIME_WAIT reaper filled the ~16k-port pool and took every outbound connection (incl. RAVEN's cameras) down. The first **continuous-metric** probe: the agent reports raw metrics, and **CP scores red/yellow/green at ingest** from operator-tunable thresholds in `cp_settings` (`host_pressure.*`, defaults warn 40% / crit 60% / CLOSE_WAIT 100 / 400), editable at staff-only `GET/PUT /settings/host-pressure`. Red fires the standard fleet alert; yellow is dashboard-only. Contrast the boolean probes (auto-login, GUI session, …) which the agent scores. |
| **Service-status report** | A typed message the agent publishes to `devices/{id}/service-status` every 5 minutes (Phase 2). Carries the observed `running | stopped | unknown` state of every service in the agent's per-device `service_allow_list`. cp-ingest persists rows into `device_services`; the dashboard's per-device Services panel renders them. Wire type: `internal/protocol/servicestatus.Report`. |
| **Allow-list (service)** | The per-device list of launchd/systemd unit names the agent reports status for. Configured in the agent's JSON config (`service_allow_list`). Starts from the install module's default; an operator can override it from the dashboard (see *Service allow-list override*). |
| **Service allow-list override** | The per-device customisation of the agent's allow-list + reporting cadence stored on `devices.service_allow_list_override` + `devices.service_status_interval_override`. Set via `PUT /devices/{id}/service-config` (cp publishes `config.update` on `devices/{id}/cmd`; agent hot-reloads its collector + publisher and ACKs on `cmd-result`). `null` override means "use the install default"; explicit `[]` override means "track nothing". The dashboard's Services Card shows `(overridden)` vs `(default)` derived from this. Phase 2 slice 2. |
| **`config.update`** | The first downward CP→agent cmd type (Phase 2 slice 2). Strictly scoped to `service_allow_list` + `service_status_interval` by the agent's `internal/handlers/configupdate.Handler`. Unsigned in Phase 2 (ADR-028); Phase 3 wraps the same envelope in the signed shape. |
| **`log.tail`** | The on-demand-log-fetch CP→agent cmd type (Phase 2 slice 3). Operator picks a logical log name + line count via the dashboard; agent resolves the name against a per-OS allow-list (`agent.PerOSAllowList()`), fetches via the entry's *Log allow-list kind* (file tail or `docker logs`), returns content in the cmd-result envelope (200 KB content cap, truncation reported). CP persists each request in `device_log_tails` keyed by `correlation_id`. Unsigned per ADR-028 (extended). |
| **Log allow-list kind** | The per-entry discriminator (`file` or `docker`) on each row of the agent's `log.tail` allow-list — `logtail.Entry{Name, Kind, Target, Label}` (issue #7 / ADR-030 § 5). `file` entries tail an absolute path with `agent.TailFile`; `docker` entries shell out to `docker logs --tail N <Target>` via `agent.TailDocker`. The wire format of the `log.tail` cmd is kind-agnostic (still `{log_name, lines}`); only the agent's resolver branches. Default Mac allow-list: 7 `file` entries + 1 `docker` entry (`plate-recognizer` → container `plate-recognizer-stream`). |
| **`device_log_tails`** | The per-request row store for log-tail flows. PK `correlation_id`; FK to `devices` with `ON DELETE CASCADE`; status walks `pending → done | error`; content stays NULL until the agent's ACK lands. Rows are reaped at 24h by `cp-ingest`'s `LogTailSweeper`. |
| **`device_services`** | The per-(device, service_name) Postgres table holding the latest reported state. PK `(device_id, service_name)`; ON DELETE CASCADE clears rows when a device is decommissioned. State column is `text` (not enum) so Phase 3 can add `failed` without a migration. |
| **Fleet alerts (Overview)** | The fleet-wide roll-up of unhealthy signals shown on the dashboard Overview page (#21). Served by `GET /fleet/alerts` (site-scoped via `ScopedDeviceQuery`): devices currently **red** or **yellow** per health probe and **stopped** per service, grouped by probe/service type with the affected device ids inline for drill-down. **Alert-only** — a type with no affected devices contributes no entry, so the page stays quiet when the fleet is healthy. This is the fleet-aggregate view the health-probes PRD deferred to slice 2. The **warn tier** is the yellow grouping, surfaced separately from red. |
| **Online threshold** | The freshness window for `last_seen` that makes a device count as "online" in the dashboard. Phase 1 value: 90 seconds (3× heartbeat interval). |
| **Desired agent version** | The agent version the Control Plane *intends* a device to run, as opposed to the version it currently reports. Agent fleet-update is modelled as setting this per device on a target set; the device reconciles toward it and CP re-pushes until reported matches desired. The source of truth for updates — there is deliberately no separate "update campaign" entity; a rollout is just desired-vs-reported across the fleet. Proposed (not yet built); design in `.scratch/agent-fleet-update/`, refining ADR-013. |
| **Agent rollback** | The automatic revert of a freshly-updated agent to its last-known-good binary when the new one fails to prove itself **alive and controllable** (mTLS connected + subscribed to its cmd topic + one heartbeat) within 5 minutes of the swap. Owned by the device-side update supervisor, not the new binary itself (which can't be trusted if broken). ADR-013's core safety guarantee. |
| **Presence** | The online/offline state of a device, stored as `devices.is_online` and maintained by `cp-ingest`: a heartbeat or an IoT Core `connected` event sets it online; an IoT Core `disconnected` event, or the sweeper finding `last_seen` older than the online threshold, sets it offline. |
| **Recovery code** | One of ten single-use codes issued to an operator at TOTP enrollment time. Shown once, stored hashed, used to recover access if the TOTP device is lost. |
| **First-run admin** | The bootstrap pattern by which the very first operator account is created. The dashboard exposes an account-creation flow at `/auth/first-run` if and only if no users exist in the database. After the first admin is created, the flow is permanently disabled (a `system_initialized` flag flips). |
